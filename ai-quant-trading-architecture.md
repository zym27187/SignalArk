# AI 量化交易系统架构 v2（个人项目版）

## 项目定位

**目标**：构建一个适合个人开发和长期迭代的 AI 量化交易系统，先保证研究结果可复现、实盘状态可控、风险边界清晰，再逐步引入 AI 增强收益。

**适用场景**：
- 单人开发和运维
- 单账户起步
- 单交易所优先
- 先做中低频，再考虑复杂策略
- 先做小资金实盘，再逐步扩大

**默认假设**：
- 第一阶段优先支持 `1 个中心化交易所`
- 第一阶段优先支持 `现货` 或 `永续合约` 二选一
- 第一阶段优先支持 `分钟级到小时级` 策略
- AI 在早期作为 `信号增强器`，而不是全自动黑盒决策核心

**第一版必须收敛的边界**：
- 只做 `1 个交易账户`
- 只允许 `1 个 active trader` 持有下单权限
- 只做 `1-3 个交易对`
- 只做 `1 个主策略`，其他策略先不并行实盘
- 第一版实时输入默认以 `BarEvent` 为主，不把 `Tick` 和 `OrderBook` 作为阻塞项
- 第一版先跑稳 `paper`，再进入 `small capital live`
- 控制面先满足 `查询 + 开关 + 告警 + kill switch`，不追求完整 dashboard

---

## 设计原则

### 1. 先做对，再做大
- 优先保证订单状态、持仓状态、账户资金状态一致
- 优先保证回测、模拟盘、实盘共用同一套事件语义
- 优先保证任何交易动作都可审计、可回放、可解释

### 2. 先模块化单体，再考虑微服务
- 个人项目初期不建议上来就拆很多服务
- 推荐采用 `模块化单体 + 少量后台进程` 的结构
- 真正需要解耦的地方只保留在 `行情接入`、`交易运行时`、`研究训练`、`控制面`

### 3. AI 是交易系统的插件，不是交易系统本身
- AI 模型负责输出概率、分数、排序、风险提示
- 是否下单、下多少、何时退出，仍应经过显式风控和执行规则
- 训练和实验环境必须与实盘环境隔离

### 4. 默认安全
- 任一关键状态不确定时，默认不下单
- 对账失败、行情延迟、时钟漂移、风控服务异常时，默认进入保护模式
- 所有策略必须接受统一的风险闸门

### 5. 少组件、强边界
- 能用 `PostgreSQL + Redis + Docker Compose` 解决的，不急着引入更重基础设施
- 能在代码边界解决的问题，不急着通过额外中间件解决

### 6. 先闭环，再补细节
- 第一版先把 `BarEvent -> Signal -> Risk -> OrderIntent -> OMS -> Paper Execution -> Position/Balance` 跑通
- `Tick / OrderBook / 多模型 / 多策略组合 / 复杂执行算法` 都不是第一版阻塞项
- 能先通过简单流程、人工介入和保守规则兜底的地方，不急着一次性自动化到极致

---

## 非目标

以下能力不应作为第一阶段目标：
- 多租户 SaaS 平台
- 高频交易或亚毫秒级低延迟系统
- 多机房、多地域高可用
- 多交易所同时实盘
- 一开始就上强化学习实盘
- 一开始就上 Kubernetes、Kafka、ELK 全家桶

## v2 总体架构

v2 采用三平面架构：
- `Research Plane`：研究、特征、训练、回测、实验管理
- `Trading Plane`：行情、策略运行时、风控、订单管理、执行、对账
- `Control Plane`：配置、调度、监控、告警、人工干预

## 核心运行链路

实盘主链路必须是闭环，而不是单向分层：

```text
行情 / 账户事件
    -> 归一化事件
    -> 策略运行时
    -> 生成 Signal
    -> Pre-Trade Risk Check
    -> 生成 OrderIntent
    -> OMS 持久化
    -> Execution Gateway 下单
    -> 交易所 ACK / Fill / Reject
    -> Position & Balance Update
    -> Reconciliation
    -> PnL / Risk / Alert
```

这条链路里的关键约束：
- `Signal` 不是订单，必须经过风控和仓位 sizing
- `OrderIntent` 必须先落库，再执行
- 订单、成交、持仓、余额必须有统一状态机
- 交易所推送和轮询结果要统一汇总到同一套状态更新逻辑

## 运行时部署建议

对于个人项目，推荐最开始只保留 4 个运行单元：

| 运行单元 | 作用 | 推荐形态 |
|----------|------|----------|
| `api` | 配置、查询、手工控制、健康检查 | FastAPI 服务 |
| `trader` | 策略运行时、风控、OMS、执行 | 单独进程 |
| `collector` | 行情采集、历史补数、数据归一化 | 单独进程 |
| `research` | 回测、训练、实验、报表 | Notebook / Job Runner |

补充说明：
- 初期不建议把 `风控`、`OMS`、`执行` 拆成多个远程服务
- 对同一个交易账户，推荐只允许 `1 个 active trader` 实例持有下单权限
- `dashboard` 可以先挂在 `api` 后面，不急着独立部署

## 第一版架构收敛建议

三平面架构是很好的认知边界，但对于个人项目，它首先应该体现在 `代码边界` 和 `权限边界` 上，而不是一开始就做成复杂的分布式部署。

第一版真正必须落地的部分：

- `Trading Plane`：标准事件模型、OMS、持仓/余额状态、paper execution、基础 pre-trade risk
- `Control Plane`：健康检查、持仓查询、策略启停、`kill switch`、Telegram 告警
- `Research Plane`：历史数据准备、最小事件驱动回测、基线策略验证

第一版可以明确延后的部分：

- `TickEvent`、`OrderBookEvent` 驱动的复杂实时决策
- `Redis Streams`、复杂消息拓扑和大规模异步解耦
- 完整模型注册中心和对象存储平台化
- 完整 Prometheus / Grafana 大盘
- 多策略组合、多账户、多交易所同时运行

如果第一版无法稳定跑通 `paper trading` 闭环，就不应该继续扩展 AI、特征工程或多策略能力。

## 核心模块设计

### 1. Market Gateway

**职责**：接入交易所行情与账户事件，并做统一归一化。

**输入**：
- WebSocket 行情
- REST 历史数据
- 账户余额
- 订单状态
- 成交回报

**输出**：
- `BarEvent`
- `TickEvent`
- `OrderBookEvent`
- `BalanceEvent`
- `OrderUpdateEvent`
- `FillEvent`

**关键要求**：
- 所有事件必须带 `event_time` 和 `ingest_time`
- 统一 `symbol`、精度、合约乘数、最小下单量
- 行情断流后必须自动重连和补数
- 任何来自交易所的原始 payload 建议保留一份原始记录

**个人项目 MVP 建议**：
- 第一版以 `BarEvent` 为主，`TickEvent` 和 `OrderBookEvent` 先保留概念，不必同时实现
- 账户事件优先保证 `BalanceEvent`、`OrderUpdateEvent`、`FillEvent`
- 如果交易所 WebSocket 质量一般，宁可加强补数和恢复，也不要先追求过细粒度行情

### 2. Event Bus

**职责**：在系统内部传递标准化事件。

**推荐实现**：
- 进程内事件：`asyncio queue / internal dispatcher`
- 跨进程轻量事件流：`Redis Streams`
- 审计与重放依据：`PostgreSQL event log`

**关键要求**：
- 不追求一开始就引入 Kafka
- 事件流必须可追踪、可重放、可定位丢失

**个人项目 MVP 建议**：
- 默认先用 `asyncio queue / internal dispatcher`
- 只有在 `collector` 与 `trader` 明确分进程并出现通信压力时，再引入 `Redis Streams`
- 第一版优先记录关键交易事件和账户事件，避免为了“全量事件平台化”过早增加复杂度

### 3. Strategy Runtime

**职责**：消费标准化市场事件，生成信号。

**职责边界**：
- 策略只负责“是否有 alpha”
- 仓位 sizing 可以由策略提供建议，但最终由风险引擎裁决
- AI 推理逻辑放在这里，以插件或模型服务适配器形式接入

**关键要求**：
- 同一套策略接口用于回测、模拟盘、实盘
- 每次策略决策必须记录输入快照、输出信号和原因摘要
- 信号建议包含置信度、有效期、目标仓位或方向，而不是直接下单

**个人项目 MVP 建议**：
- 第一版先支持 `1 个 active strategy`
- 不需要一开始就做复杂的策略热插拔和策略编排系统
- AI 推理接口先留边界即可，不应成为交易骨架落地的前置条件

### 4. Risk Engine

**职责**：对信号和订单进行统一风险约束。

**风险层级**：
- `strategy-level`：单策略限额
- `account-level`：账户总风险敞口
- `exchange-level`：交易所接口和规则限制
- `global-level`：全局熔断和 kill switch

**风险阶段**：
- `pre-trade`：下单前检查
- `in-trade`：持仓过程控制
- `post-trade`：交易后监控和复盘

**典型检查**：
- 最大仓位
- 最大杠杆
- 当日最大亏损
- 单策略最大回撤
- 单品种最大敞口
- 流动性阈值
- 滑点估计上限
- 重复信号和重复下单防护

**个人项目 MVP 建议**：
- 第一版必须先做稳 `pre-trade risk`
- `in-trade` 和 `post-trade` 可以先从少量硬规则开始，例如日亏上限、只减仓模式、策略暂停后保留平仓能力
- 风控规则应优先追求可解释和保守，而不是复杂自适应

### 5. OMS / Portfolio State

**职责**：管理订单状态、持仓状态、账户资金状态，是交易域的核心事实源。

**必须持久化的对象**：
- `Signal`
- `OrderIntent`
- `Order`
- `Fill`
- `Position`
- `BalanceSnapshot`
- `PnLSnapshot`

**关键要求**：
- 订单必须有明确状态机：`NEW -> ACK -> PARTIALLY_FILLED -> FILLED / CANCELED / REJECTED`
- 所有外部下单请求都要有幂等键
- 本地状态和交易所状态不一致时，必须触发对账流程
- 持仓和余额更新不能只依赖内存

**个人项目 MVP 建议**：
- 这是第一版最值得优先完成的模块之一
- 宁可先少做执行花样，也不要牺牲状态一致性
- 只要 OMS 和持仓状态不稳，paper 和 live 都不应继续扩功能

### 6. Execution Gateway

**职责**：把内部订单意图转换为交易所订单，并处理回报。

**建议设计**：
- 通过 `exchange adapter` 封装交易所差异
- 对热路径保留原生接口能力，不把交易逻辑锁死在第三方库上
- 允许 `paper`、`sandbox`、`live` 三种执行模式共用接口

**关键要求**：
- 下单前先做交易所规则校验
- 订单失败必须带明确失败原因
- 对市价单和限价单分别配置滑点和超时策略
- 提供 `cancel all`、`reduce only`、`panic close` 这类保护指令

**个人项目 MVP 建议**：
- 第一版先实现 `paper` 执行适配器，并用它跑通闭环
- `sandbox` 不是必需项，有些交易所的测试环境并不可靠
- 真实 `live` 下单应该晚于 paper、回测一致性和基础对账

### 7. Reconciliation

**职责**：对账，修复系统状态漂移。

**对账对象**：
- 订单状态
- 成交记录
- 持仓
- 余额
- 已实现 / 未实现 PnL

**触发条件**：
- 定时对账
- 启动恢复
- WebSocket 断线重连
- 订单异常
- 手工干预后

**关键要求**：
- 对账异常时默认降级为保护模式
- 对账流程必须可追踪
- 明确“修复动作”与“观测动作”的边界，避免自动修错越修越乱

**个人项目 MVP 建议**：
- 第一版先实现“发现漂移 + 告警 + 进入保护模式”
- 自动修复可以后置，并严格限制在少量可验证场景
- 对账模块的目标首先是防止带病运行，而不是追求全自动愈合

### 8. Research & Backtest

**职责**：研究策略、训练模型、验证收益和风险。

**关键要求**：
- 回测必须与实盘共享事件模型和订单语义
- 必须考虑手续费、滑点、流动性、撮合延迟
- 必须避免未来函数和特征泄露
- 对 AI 策略建议采用 walk-forward 验证

**个人项目 MVP 建议**：
- 第一版只需要支持 `K 线驱动 + 基线策略 + 标准绩效报表`
- 不必一开始就建设完整实验平台或大规模参数搜索系统
- 回测系统的首要价值是验证交易语义一致性，而不是追求花哨的研究能力

### 9. Monitoring & Control

**职责**：监控系统状态、交易状态、风险状态，并支持人工干预。

**监控维度**：
- 服务健康
- 行情延迟
- 订单成功率
- 风险指标
- 策略表现
- 对账状态
- 模型版本与推理延迟

**人工控制能力**：
- 启停策略
- 切换 paper/live
- 手动冻结某个 symbol
- 全局 kill switch
- 调整策略参数和风险参数

**个人项目 MVP 建议**：
- 第一版优先用 `结构化日志 + Telegram 告警 + 少量 API`
- `Prometheus / Grafana` 可以在稳定运行后再补
- 先确保异常能被发现、能被人为接管，而不是一开始就追求大盘完备

## 研究平面与交易平面分离

这是 v2 里最重要的调整之一。

### Research Plane

负责：
- 数据准备
- 特征计算
- 训练
- 参数搜索
- 回测
- 模型评估
- 生成报告

特点：
- 可用批处理
- 允许重计算
- 允许使用较重依赖
- 不直接持有实盘下单权限

### Trading Plane

负责：
- 实时事件处理
- 实时推理
- 风控裁决
- 下单和撤单
- 持仓更新
- 对账

特点：
- 面向低延迟和高稳定
- 只保留必要依赖
- 配置和模型变更要可回滚
- 任何异常默认偏保守

### 结论

训练、调参、实验追踪不应该出现在实盘热路径里。  
在线交易运行时只保留以下能力：
- 加载已发布模型
- 执行在线特征变换
- 调用推理
- 记录推理输入输出
- 做版本切换和回滚

对于个人项目，这个“分离”首先意味着：
- 同一仓库内保持清晰模块边界
- 研究代码默认不持有实盘权限
- 不必为了“平面分离”而过早拆成多套复杂服务

## 数据架构 v2

个人项目不建议同时维护多套数据库来承载同一类核心数据。  
推荐从下面的组合开始：

| 数据类型 | 推荐方案 | 用途 |
|----------|----------|------|
| 关系型 + 时序核心数据 | PostgreSQL，TimescaleDB 可在数据量上来后启用 | K 线、订单、成交、持仓、余额、PnL、策略事件 |
| 缓存和轻量事件流 | Redis，可按需引入 | 行情缓存、去重、分布式锁、轻量消息流 |
| 模型和报告产物 | 本地磁盘优先，MinIO 后续可选 | 模型文件、回测报告、训练日志、数据快照 |

如果第一版只做单机 `paper` 或小规模 `live`，一个更轻的起步组合通常就够了：

- `PostgreSQL`：核心交易状态和标准化数据
- `本地磁盘`：模型文件、报表、数据快照
- `Redis`：只在缓存、锁或跨进程通信需求明确后再引入

### 数据分层

建议明确 4 层数据：

#### 1. Raw Layer
- 原始交易所响应
- 原始新闻、情绪、链上数据
- 保留原始字段，便于回放和排错

个人项目第一版通常只需要保留交易所原始响应；新闻、情绪、链上数据都应后置。

#### 2. Normalized Layer
- 标准化后的 K 线、tick、订单簿、账户事件
- 统一 symbol、时区、精度、字段命名

个人项目第一版可优先围绕标准化 `K 线 + 账户事件` 建模。

#### 3. Feature Layer
- 技术指标
- 统计特征
- 因子特征
- 模型输入快照

#### 4. Trading State Layer
- 信号
- 订单意图
- 订单状态
- 成交
- 持仓
- 余额
- 风险事件

### 数据字段规范

每类事件尽量统一带上：
- `event_id`
- `source`
- `exchange`
- `account_id`
- `symbol`
- `event_time`
- `ingest_time`
- `strategy_id`
- `model_version`

## AI/ML 设计策略

### AI 在系统中的位置

AI 不应该直接绕过交易系统去下单。推荐的方式是：

```text
Market / Feature Data
    -> Model Inference
    -> Alpha Score / Direction / Confidence
    -> Strategy Rules
    -> Risk Engine
    -> OrderIntent
```

推荐让模型输出：
- 未来一段时间收益概率
- 多空方向分数
- 波动率预测
- regime classification
- 是否降低仓位的风险提示

不建议一开始就让模型直接输出：
- 原始下单指令
- 任意杠杆倍数
- 黑盒全流程交易动作

### 模型落地顺序

个人项目推荐按以下顺序推进：

1. 规则策略基线  
   先有稳定可解释的均线、动量、均值回归或因子策略。

2. 传统 ML  
   优先 `LightGBM / XGBoost / Logistic Regression`，通常更容易解释和迭代。

3. 时序深度模型  
   在 baseline 已经稳定后，再尝试 `LSTM / TCN / Transformer`。

4. 强化学习  
   仅在你已经有可信的撮合模拟器、对账体系和稳定基线后再进入。

### 模型治理

必须记录：
- 训练数据快照版本
- 特征 schema hash
- 标签定义
- 训练代码版本
- 模型版本
- 回测结果
- 上线时间
- 回滚记录

推荐上线流程：
- `offline backtest`
- `paper trading`
- `shadow mode`
- `small capital live`
- `gradual scale-up`

## 风控体系 v2

### Pre-Trade Risk

下单前检查：
- 最大仓位
- 最大名义价值
- 最大杠杆
- symbol 冷却时间
- 价格偏离阈值
- 预计滑点
- 最小流动性
- 重复订单
- 模型版本是否允许交易
- 行情是否过期

### In-Trade Risk

持仓中控制：
- 固定止损
- 追踪止损
- 时间止损
- 仓位再平衡
- 动态降杠杆
- 只减仓模式
- 某策略暂停但保留平仓能力

### Post-Trade Risk

交易后检查：
- 当日 PnL
- 连续亏损次数
- 策略回撤
- 账户风险暴露
- 实盘与回测偏差
- 实际滑点与预估滑点偏差

### 熔断与保护模式

建议的保护触发条件：
- 行情延迟超过阈值
- 关键服务心跳丢失
- 本地持仓与交易所持仓不一致
- 日内亏损超过阈值
- 某 symbol 出现连续拒单
- 对账异常未恢复

进入保护模式后的动作建议：
- 禁止开新仓
- 保留减仓和平仓
- 推送高优先级告警
- 记录完整诊断信息

---

## 回测、模拟盘、实盘一致性

这是交易系统最容易做坏的地方。

### 必须统一的内容

- 事件类型
- 策略接口
- 订单类型语义
- 手续费模型
- 滑点模型
- 撮合逻辑
- 持仓计算逻辑
- PnL 计算逻辑

### 建议实践

- 策略运行时不要区分“回测版”和“实盘版”两套逻辑
- 差异只应体现在 `data source` 和 `execution adapter`
- 回测报告必须能追溯到策略版本、数据版本和参数版本
- 对 AI 策略使用时间序列交叉验证和 walk-forward

### 不建议的做法

- 回测用 K 线，实盘却靠 tick 做完全不同决策
- 回测按收盘价成交，实盘却忽略滑点和流动性
- 训练特征和在线特征由两套代码分别实现

## 技术选型建议

### 推荐选型

| 领域 | 推荐方案 | 理由 |
|------|----------|------|
| 主语言 | Python 3.11+ | 生态成熟，研究和交易可共用 |
| API | FastAPI | 轻量、类型友好、易运维 |
| 配置 | Pydantic Settings + YAML | 配置分层清晰 |
| ORM / Migration | SQLAlchemy + Alembic | 交易状态建模稳健 |
| 核心数据库 | PostgreSQL，TimescaleDB 可选 | 第一版先把状态和 schema 做稳 |
| 缓存 / 轻量消息 | Redis，按需引入 | 不必为了“标准架构”强行提前接入 |
| 研究数据处理 | Pandas / Polars | 兼顾易用和性能 |
| 模型训练 | LightGBM / XGBoost / PyTorch | 先轻后重 |
| 回测 | 自定义事件驱动引擎 | 与实盘语义对齐 |
| 监控 | 结构化日志 + Telegram，Prometheus/Grafana 后续可选 | 第一版优先可观测和可告警 |
| 部署 | Docker Compose | 适合单机或单 VPS |

### 建议延后引入

| 技术 | 为什么延后 |
|------|------------|
| Kubernetes | 运维复杂度远高于当前收益 |
| Kafka | 个人项目早期吞吐通常不需要 |
| ELK | 日志系统偏重，可先用 JSON 日志 + Grafana 栈 |
| 多交易所并行实盘 | 状态复杂度会上升一个量级 |
| DEX 交易聚合 | 链上执行和风控复杂很多 |
| 强化学习实盘 | 需要更强模拟环境和更长验证周期 |

## 部署架构建议

### 开发环境

建议最小化为：
- `postgres`
- `api`
- `trader`
- `collector`
- `research` 或 notebook

可选：
- `redis`

说明：
- 如果 `collector` 与 `trader` 先不拆复杂通信，开发环境完全可以不先上 Redis

### 生产环境

个人项目推荐先用：
- `1 台 VPS / 云主机 / 本地稳定主机`
- `Docker Compose + systemd`
- `结构化日志 + Telegram 告警`
- 数据库定时备份

在进入长期稳定运行后，可再补：
- `Prometheus`
- `Grafana`

### 为什么不先上 Kubernetes

对于个人项目，最稀缺的不是容器编排能力，而是：
- 稳定的交易状态
- 可追溯的事件流
- 有效的策略迭代速度
- 可操作的风控流程

Kubernetes 不是不能上，而是不应成为第一阶段瓶颈。

## 推荐实施路线图

### Phase 0：定边界

先明确这些决策：
1. 先做哪个交易所
2. 先做现货还是永续
3. 先做哪些交易对
4. 先做什么频率的策略
5. 允许什么风险上限

### Phase 1：交易骨架先跑通

目标：
- 行情接入
- 标准事件模型
- OMS / Position State
- 基础风控
- Paper Trading
- Telegram 告警
- 默认只围绕 `BarEvent` 跑通第一版闭环

完成标准：
- 可以稳定接行情
- 可以记录信号、订单、成交、持仓
- 可以手动启停策略
- 可以跑完整的模拟交易闭环

### Phase 2：研究与回测统一

目标：
- 自定义事件驱动回测引擎
- 与实盘共用策略接口
- 手续费、滑点、延迟建模
- 基线策略验证

完成标准：
- 同一策略可在回测和 paper 模式运行
- 回测结果可复现
- 能导出标准绩效报表

### Phase 3：引入 AI 增强

目标：
- 特征工程 pipeline
- 传统 ML 模型
- 模型版本管理
- 在线推理接入策略运行时

完成标准：
- AI 输出 alpha score 或 regime signal
- 可以记录模型版本和推理结果
- 先在 paper/shadow 模式验证

### Phase 4：小资金实盘

目标：
- 小仓位 live trading
- 对账与异常恢复
- 保护模式和 kill switch
- 实盘日报和风险告警

完成标准：
- 遇到异常可自动进入保护模式
- 订单、持仓、余额可定时对账
- 能稳定运行一段时间且无状态漂移

### Phase 5：逐步扩展

可考虑逐步增加：
- 多策略组合
- 多品种
- 多模型 ensemble
- 更细粒度执行逻辑
- 更多数据源

这一步之前，不建议同时扩多交易所和多策略。

## v2 核心结论

这套 v2 的核心不是“功能更多”，而是“更适合个人项目落地”：

- 架构从“多层罗列”改成“交易闭环优先”
- 从“AI 是中心”改成“交易系统是中心，AI 是增强器”
- 从“多组件堆叠”改成“少组件、强状态、一致语义”
- 从“先上大而全”改成“先跑通可审计的 paper/live skeleton”

如果按这个方向推进，第一版最值得优先做的不是复杂模型，而是：
1. 标准事件模型
2. OMS / 持仓 / 余额状态机
3. 风控和保护模式
4. 回测与实盘一致性
5. 基线策略
6. 然后再接 AI

如果还要再压缩成一句更适合个人项目执行的话，就是：

先把 `单交易所、单账户、单策略、K 线驱动、paper 闭环` 做稳，再谈 AI、扩品种和实盘放量。

---

**备注**：对于个人 AI 量化项目，最优路径通常不是“最先进的模型”，而是“最稳的交易内核 + 可迭代的研究流程”。v2 就是按这个方向收敛后的架构版本。
