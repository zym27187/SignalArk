# V1 测试规范与执行手段

这份文档定义 V1 阶段统一的测试规范。  
所有 phase 文件和所有 AI 执行提示词都应以本文件为准。

---

## 1. 测试目标

V1 不是追求测试数量，而是确保下面三件事成立：

1. 核心交易语义是稳定的
2. 关键状态变化是可验证的
3. 重要失败路径不会被忽略

对于这个项目，测试首先要保护的是：

- 事件模型
- 订单状态机
- 风控放行与拒绝
- `OrderIntent -> Order -> Fill -> Position / Balance` 主链路
- 对账与保护模式

---

## 2. 测试分层

V1 默认采用三层测试：

### 2.1 `unit`

用于验证纯逻辑，不依赖外部服务。

典型内容：

- 数据模型校验
- 枚举与状态机迁移
- 风控规则判断
- PnL / 持仓计算
- 事件标准化函数
- 策略信号生成逻辑

要求：

- 应尽量快
- 应可重复执行
- 不依赖网络
- 不依赖真实交易所

### 2.2 `integration`

用于验证模块协作，允许依赖数据库或本地运行服务。

典型内容：

- repository / DAO 读写
- migration 与表结构
- collector 到标准事件的转换
- trader 主循环与事件分发
- OMS 与 paper execution 衔接
- API 与控制动作
- 对账与保护模式任务

要求：

- 允许依赖 `PostgreSQL`
- 默认不依赖真实网络
- 外部交易所返回应优先使用 fixture / mock / 录制样本

### 2.3 `e2e`

用于验证最小闭环，只覆盖少量关键路径。

V1 中只建议保留少量 e2e：

- 基线策略 -> Signal -> 风控 -> OMS -> paper fill -> 持仓更新
- `kill switch` 后禁止继续开仓
- 对账异常触发保护模式

要求：

- 数量要少
- 覆盖必须关键
- 运行时间允许稍慢，但不能无限膨胀

---

## 3. V1 不建议的测试方式

V1 阶段不建议默认依赖下面这些方式：

- 真实交易所在线测试作为日常测试主路径
- 需要手工修改大量环境后才能跑的测试
- 随机性很强、重复执行结果不稳定的测试
- 与业务无关但维护成本很高的 UI 测试体系
- 为了测试而引入过重基础设施

真实交易所联调应视为 `人工 smoke check`，不应作为 V1 常规自动测试主干。

---

## 4. 推荐目录与命名

推荐目录：

```text
tests/
  unit/
  integration/
  e2e/
```

推荐命名：

- `test_order_state_machine.py`
- `test_risk_rules.py`
- `test_bar_normalization.py`
- `test_paper_execution_flow.py`
- `test_kill_switch.py`

推荐原则：

- 文件名表达被测能力
- 一个测试文件只聚焦一个能力域
- 不要把多个 phase 的大杂烩塞进同一个测试文件

---

## 5. 推荐工具与执行方式

V1 推荐默认采用 `pytest`。

推荐执行方式：

```bash
pytest tests/unit -q
pytest tests/integration -q
pytest tests/e2e -q
```

如果项目后续增加 marker，可采用类似方式：

```bash
pytest -m unit -q
pytest -m integration -q
pytest -m e2e -q
```

如果某个阶段只改了纯逻辑代码，至少应运行相关 `unit` 测试。  
如果某个阶段引入了数据库、事件流、API、paper execution 或对账逻辑，应同时运行相关 `integration` 测试。  
如果改动影响完整交易闭环，应补充最小 `e2e` 验证。

---

## 6. 测试数据策略

### 6.1 市场数据

默认使用固定 fixture 或录制样本：

- K 线样本
- 标准化前交易所 payload
- 成交和订单更新样本

不要让常规测试依赖实时外部行情。

### 6.2 时间

涉及时间的测试应尽量固定：

- 固定 `event_time`
- 固定 `ingest_time`
- 固定时区

避免因为当前时间变化导致测试不稳定。

### 6.3 随机性

如果使用随机值：

- 必须设置固定 seed
- 或完全移除随机依赖

---

## 7. 数据库测试手段

涉及数据库时，优先使用下面方式之一：

1. 本地临时测试库
2. Docker Compose 中的独立测试数据库
3. 每次测试前后清理表数据

要求：

- 测试之间应尽量相互隔离
- 不要把开发数据和测试数据混在一起
- migration 测试要能验证建表是否成功

如果当前环境没有测试数据库，AI 需要：

- 明确说明未运行集成测试的原因
- 至少补齐对应单元测试
- 在交付说明里写出缺失的验证风险

---

## 8. Mock / Stub / Fixture 原则

推荐做法：

- 纯逻辑用 unit test
- 外部交易所响应用 fixture 或 adapter mock
- paper execution 用可预测的 stub / fake
- Telegram 告警用 mock sender

不推荐：

- 为了“看起来真实”而让测试直接打外部 API
- 大量脆弱 mock 覆盖真实业务逻辑

原则是：

> 尽量 mock 边界，不要 mock 核心交易语义。

---

## 9. 每次 AI 交付的测试汇报格式

AI 在完成任一 phase 或子任务后，测试汇报至少要包含：

1. 运行了哪些测试
2. 哪些测试通过
3. 哪些测试没有运行
4. 为什么没运行
5. 当前剩余测试风险是什么

推荐格式：

```text
测试情况：
- 已运行：pytest tests/unit/...
- 已运行：pytest tests/integration/...
- 未运行：tests/e2e/...
- 原因：当前阶段尚未形成完整闭环 / 本地缺少测试数据库
- 风险：订单恢复流程仍缺少集成验证
```

## 10. 各阶段最低测试门槛

### Phase 0

- 至少 1 个配置加载测试

### Phase 1

- 至少 1 组订单状态机单元测试
- 至少 1 组核心对象构造 / 校验测试

### Phase 2

- 至少 1 组 migration / 建表测试
- 至少 1 组 repository 读写测试
- 至少 1 组幂等更新测试

### Phase 3

- 至少 1 组 bar 标准化测试
- 至少 1 组历史数据转 `BarEvent` 测试
- 至少 1 组恢复或补数路径验证

### Phase 4

- 至少 1 组事件分发测试
- 至少 1 组 trader 消费 bar 事件测试

### Phase 5 / 5A / 5B / 5C

- 至少 1 组 `Signal -> OrderIntent -> Order` 测试
- 至少 1 组 paper fill / reject / cancel 场景测试
- 至少 1 组持仓、余额或基础 PnL 更新测试

### Phase 6 / 6A / 6B / 6C

- 至少 1 组风险放行 / 拒绝测试
- 至少 1 组 `kill switch` 或策略暂停测试
- 至少 1 次告警或关键异常日志验证

### Phase 7

- 至少 1 组策略信号单元测试
- 至少 1 条从策略到订单的端到端路径

### Phase 8

- 至少 1 组回测事件驱动流程测试
- 至少 1 组 paper / backtest 接口一致性验证

### Phase 9

- 至少 1 组恢复或定时对账测试
- 至少 1 组异常进入保护模式测试
- 至少 1 组漂移检测测试

---

## 11. V1 的测试完成定义

V1 不能只靠“代码写完”判断完成。  
至少应满足：

- 关键领域逻辑有 unit test
- 关键模块协作有 integration test
- 最小交易闭环有少量 e2e
- AI 在每次交付中明确说明测试覆盖和剩余风险

一句话说：

> V1 的测试目标不是追求覆盖率数字，而是确保最关键的交易闭环和失败路径都被验证过。
