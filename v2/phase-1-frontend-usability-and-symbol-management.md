# Phase 1：前端易用性与股票代码管理

这份文件用于把 V2 的前端目标从“能显示数据”推进到“金融小白也能理解和操作”。

## 本次目标

重构前端信息表达方式，并补齐股票代码添加入口，让用户可以更直观地理解系统状态、风险提示和股票代码当前的生效层级。

## 本次必须完成的任务

- 优化主要页面的信息架构，让关键页面遵循“先结论、再解释、再细节”。
- 为关键术语提供稳定解释方式，例如：
  - 持仓
  - 成交
  - 冻结资金
  - 权益
  - 回撤
  - kill switch
  - protection mode
- 补齐空状态、异常状态、降级状态和确认提示。
- 提供前端添加股票代码的入口。
- 让股票代码输入后至少能看到：
  - 规范化后的代码
  - 股票名称或名称缺失提示
  - 市场归属
  - 格式是否合法
  - 当前属于观察、supported 还是 runtime 层
- 对会影响 trader 运行标的的动作提供显式确认和后果说明。

## 本次不要做

- 不做完整新手引导系统
- 不做复杂动画式 onboarding
- 不在本阶段直接做大规模后端配置热更新
- 不把“添加股票代码”默认实现成直接改动 trader 生效范围

## 默认实现细节

### 1. 首页或关键视图建议

建议优先让用户先看到：

- 当前系统是否可用
- 当前策略是否开启
- 当前数据是否真实或降级
- 当前账户整体状态
- 当前关注股票代码及其状态

### 2. 股票代码添加入口建议

默认建议包含：

- 输入框
- 格式校验
- 名称和市场归属反馈
- 当前层级状态展示
- 是否申请加入运行时范围的显式选项

### 3. 面向金融小白的文案原则

- 少用缩写，不省略上下文
- 数字附近尽量说明“它代表什么”
- 高风险动作先解释后确认
- 遇到异常优先告诉用户“影响是什么”

## 完成标准

- 金融小白能读懂主要页面的核心状态
- 股票代码可以通过前端被添加和校验
- 用户能理解新增股票代码当前是否只观察、已支持或已进入运行时
- 高风险动作和异常状态具备可理解的解释层

## 最低验证要求

- 至少覆盖一个主要页面的前端测试
- 至少覆盖股票代码输入、校验或状态展示的一组测试
- 至少验证一次高风险动作确认或降级状态展示
- 如引入后端只读接口，至少运行相关集成验证

## 本次交付时必须汇报

- 哪些页面完成了信息架构优化
- 关键术语如何解释
- 股票代码添加入口支持了哪些校验和状态反馈
- 哪些动作只是观察层，哪些会影响运行时
- 当前仍有哪些页面对金融小白不够友好
- 是否可以进入 Phase 2
