# SignalArk V2 范围草案

## 文档定位

这份文档用于定义 SignalArk 下一阶段迭代的范围草案。

它不是重写提案，不是详细实施计划，也不是对现有 V1 的否定；它的目标是把当前已经跑通的 V1 闭环，升级为一个更容易持续研究、持续诊断、持续扩展的个人项目版本。

本草案主要基于以下前提形成：

- 当前仓库已经具备 `collector -> strategy -> risk -> OMS -> paper execution -> persistence -> API/research` 的最小闭环。
- 当前主要缺口集中在账户资金可见性、诊断能力一致性、research 结果管理和 AI 扩展接缝，而不是基础骨架缺失。
- 历史归档文档 [archive/ai-quant-trading-architecture.md](../archive/ai-quant-trading-architecture.md) 已经提供了较成熟的 V2 架构方向，但还没有收敛到适合当前仓库直接执行的范围定义。

## 当前判断

SignalArk 的 V1 已经达到了“可继续迭代”的状态，而不是仍停留在“先把骨架搭起来”的状态。

更具体地说，当前系统已经完成了以下关键前提：

- 有稳定的 A 股 `paper trading` 主链路。
- 有最小一致性的 research/backtest 能力。
- 有基础 control plane、reconciliation 和 protection mode。
- 有前端控制台和最小运维查询入口。

因此，V2 更适合被定义为：

> 在不扩大核心业务边界的前提下，强化三平面边界、提升研究与诊断效率、为 AI 增强能力预留可控接缝的一次结构化迭代。

## V2 总目标

V2 的目标不是“功能数量翻倍”，而是让系统从“V1 闭环可跑”升级为“可持续研究、可持续诊断、可持续扩展”的版本。

一句话概括：

> 先把 `Research Plane / Trading Plane / Control Plane` 的边界、契约和观测面做清楚，再在统一语义下推进研究增强和 AI 增强。

## V2 固定边界

V2 仍然默认保持以下固定边界，不在本轮主动扩张：

- 市场仍固定为 `cn_equity`。
- 交易标的仍固定为 `a_share`。
- 执行模式仍固定为 `paper`。
- 交易账户仍固定为 `1 个`。
- trader 运行时仍只允许 `1 个 active trader` 持有下单权限。
- 实盘主策略仍保持 `1 个 active 主策略`，research 侧可以做多策略对照，但不进入并行实盘编排。
- 交易触发仍以 `closed_bar` 为核心，不把 `TickEvent` 或 `OrderBookEvent` 作为 V2 阻塞项。
- 数据源仍以当前 `eastmoney + fixture` 体系为基础，不在 V2 内主动引入多数据源编排。
- PostgreSQL 仍是核心恢复事实源，不在 V2 内引入 Kafka、Kubernetes 或分布式事件平台。
- 系统形态仍以“模块化单体 + 少量进程”为主，不做大规模服务拆分。

## V2 核心范围

### 1. 三平面边界收口

V2 的第一优先级不是增加更多模块，而是明确当前能力分别属于哪个平面，并收口共享契约。

本主题内建议纳入：

- 明确 `Research Plane`、`Trading Plane`、`Control Plane` 的职责边界。
- 盘点当前跨平面共享的数据契约，例如：
  - strategy audit
  - research manifest
  - runtime bar audit
  - balance snapshot
  - control action result
- 优先通过目录、类型定义、API 契约和输出语义来体现边界，而不是通过新的远程服务去体现边界。
- 收口“同一个事实在多个地方重复命名、重复导出、语义略有漂移”的问题。

### 2. 控制面与运维可见性增强

V2 需要把当前控制面从“能查基本状态”推进到“值守时能解释账户与运行状态”。

本主题内建议纳入：

- 增加账户资金相关只读视图：
  - 最新余额
  - 可用资金
  - 冻结资金
  - 关键资金快照时间
- 让操作员能直接理解 `现金变化 / 持仓变化 / 权益变化` 的关系，而不是只能查表或读日志。
- 补齐控制动作结果回显的完整语义，尤其是高风险动作和批量动作。
- 为降级模式提供显式提示，区分“真实接口无数据”和“fixture 示例兜底”。
- 让前端、HTTP API 和 MCP 至少在关键诊断能力上保持最小一致性。

### 3. Trading Plane 硬化与审计增强

V2 不应重写现有 trader 主链路，但需要让它更可解释、更可排查、更适合长期值守。

本主题内建议纳入：

- 把 runtime 实际消费的 bar、策略实际看到的 bar、下单实际依据的关键上下文，进一步收口为第一类审计事实。
- 强化保护模式、对账状态、控制状态和恢复状态的统一展示方式。
- 补强 drift、延迟、行情降级、控制动作副作用等状态的显式输出，而不是只留在日志里。
- 避免无提示的静默回退，优先把“不确定”状态暴露给操作员。
- 继续保持 `Signal -> OrderIntent -> OMS -> Execution -> Position/Balance` 的统一状态语义，不在 V2 内引入绕过风控或绕过 OMS 的新路径。

### 4. Research Plane 演进

V2 需要把当前 research 能力从“最小一致性验证”推进到“可对比、可复核、可迭代优化”。

本主题内建议纳入：

- 标准化 research 运行模式，例如：
  - preview
  - evaluation
  - parameter scan
  - walk-forward
- 统一 research 输出中的 manifest、样本信息、参数信息、指标信息和分段结果表达。
- 让不同策略在同一套样本、同一套成本模型、同一套指标定义下可横向比较。
- 优先提升研究结论的可复现性和可解释性，而不是先上更复杂的优化器。
- 将 research 结果与控制面、MCP 或诊断入口建立最小引用关系，避免研究和运行完全割裂。

### 5. AI 增强接缝

V2 可以引入 AI 增强，但 AI 在本轮仍然不是交易系统的中心。

本主题内建议纳入：

- 把 AI 定位为 `signal enhancer / reviewer / scorer`，而不是直接越过现有风控和执行契约。
- 为 AI 策略实验保留稳定 provider seam、版本标识、参数快照和决策审计信息。
- 保持 deterministic fallback，使本地验证和 CI 不依赖外部模型即可运行。
- 让 AI 策略与基线策略复用同一套 strategy audit、research manifest 和对照框架。
- 不在 V2 内把 AI 扩展成在线训练平台、自动调参平台或黑盒自治 trader。

## V2 明确不纳入范围

以下内容建议明确排除在本轮 V2 范围之外：

- 真实券商或实盘交易接入
- 多账户体系
- 多市场数据源
- 多策略组合实盘编排
- Tick / OrderBook 驱动的高频或超低延迟交易
- 大规模分布式消息基础设施，如 Kafka
- Kubernetes、复杂云原生编排和多地域高可用
- 完整 ML 训练平台、对象存储实验平台、模型注册中心平台化
- 为了追求“架构先进”而进行的大规模目录或服务重写

## V2 实施原则

- 以增量迭代替代推倒重来，优先复用现有 V1 主链路。
- 先统一契约和观测，再扩功能深度。
- 先让 operator 能看懂、能定位、能回放，再追求更多自动化。
- 先让 research 结果可复现、可对照，再增加更重的 AI 能力。
- 每个阶段都应能独立测试、独立验收、独立回滚。

## V2 完成线

只有当以下条件基本成立时，V2 才算进入“完成”状态：

- 三平面职责边界在代码组织和文档层面都已清晰，不再依赖口头约定。
- 账户余额、可用资金、冻结资金和权益变化可以通过控制台或 API 直接解释。
- 关键诊断能力在前端、HTTP API、MCP 三者之间达到最小一致性。
- runtime bar、research snapshot、control action result 等关键审计事实具备稳定语义。
- research 结果具有统一 manifest 和可复现样本说明，能够稳定对比 baseline 与 AI 候选策略。
- V2 仍保持单账户、单主策略、A 股 `paper` 的稳定主链路，没有因扩展而破坏 V1 核心闭环。

## 建议拆解方向

本草案建议后续把 V2 拆成以下几个阶段文档，而不是一次性并行推进：

1. `Phase 0`：V2 边界、共享契约与目录约定
2. `Phase 1`：控制面与账户资金可见性
3. `Phase 2`：诊断能力统一与降级模式表达
4. `Phase 3`：research 输出标准化与实验对照能力
5. `Phase 4`：AI 策略增强接缝与统一审计

如果要进一步压缩优先级，建议先做：

1. 账户资金可见性 + 控制面一致性
2. runtime / research / MCP 诊断契约统一
3. research 标准化输出
4. AI 增强接缝

## 待确认但不阻塞本草案的问题

- V2 是否需要把 `apps/mcp` 的能力提升到与 HTTP API 同级的诊断入口。
- V2 是否需要把部分 V1 遗留 TODO 直接吸收为 V2 Phase 1 的正式范围。
- V2 的第一阶段是否只做文档与契约收口，还是允许同步补一小批低风险只读接口。

当前建议是：

> 先把 V2 作为“收口边界并提升一致性”的迭代启动，而不是作为“开启新一轮大范围功能扩张”的借口。
