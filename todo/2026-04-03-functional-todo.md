# SignalArk 功能实现待办

生成时间：2026-04-03

## 评估范围

- 仅聚焦当前 V1 已有实现上的功能补齐与产品闭环。
- 不包含真实券商、多账户、多市场、多策略、AI/ML 训练平台等范围外事项。
- 本轮重点不是补底层脚手架，而是减少语义占位、补齐前端入口、增强操作与诊断可见性。

## 当前判断

- 交易主链路已经基本闭环：collector、strategy、risk、OMS、paper execution、persistence、control plane、reconciliation、research 都已落地。
- 当前更值得投入的工作，主要集中在“功能收口”而不是“功能起步”。
- 优先级最高的缺口，集中在 research 契约语义、运维筛选能力、控制动作结果可见性，以及资金视图补齐。

## P0：优先收口语义漂移

- [ ] 收口 research snapshot 契约，处理 `runtimePnlCurve` 与 `backtestEquityCurve` 语义重叠的问题。
- [ ] 明确 research 页对外展示的是“回测权益曲线”还是“runtime 盘中权益曲线”，避免前端和导出文件继续沿用占位字段。
- [ ] 统一 research API、research CLI 导出、前端类型定义里的同名字段语义，减少后续继续兼容占位命名的成本。

## P1：补齐已有后端能力的前端入口

- [ ] 给运维页筛选器补 `status` 维度，接通历史订单状态筛选，而不只支持 `symbol`、`trader_run_id` 和时间窗。
- [ ] 给历史成交补 `order_id` 级别定位能力，支持围绕单笔订单追踪完整 fill 结果。
- [ ] 让运维页筛选能力与后端接口参数保持一致，避免“后端已支持但前端无入口”的半闭环状态。

## P1：补齐控制动作闭环

- [ ] 给高风险控制动作补确认交互，至少覆盖 `cancel-all` 与 kill switch 类操作。
- [ ] 在控制面板展示 `cancel-all` 的执行结果统计，如请求数、成功撤单数、跳过数，而不只是返回一条 message。
- [ ] 增加最近一次操作动作的结果回显，方便值守人员确认动作是否真正落地。

## P1：补齐资金与组合可见性

- [ ] 增加“最新余额 / 可用资金 / 冻结资金”只读视图，而不只展示持仓和组合权益曲线。
- [ ] 让操作员能直接从控制台理解“现金变化、持仓变化、权益变化”三者关系，减少依赖查库。
- [ ] 评估是否补一个账户资金快照接口，复用已持久化的 `balance_snapshots` 数据。

## P2：增强 MCP 与诊断一致性

- [ ] 给 MCP 补齐 runtime bar 审计读取能力，避免 AI 侧只能看重拉行情，不能看 trader 实际消费数据。
- [ ] 给 MCP 补齐组合权益曲线读取能力，让 AI 侧可以直接基于账户组合曲线做排查和复核。
- [ ] 保持 MCP、HTTP API、前端三者在诊断能力上的最小一致性，减少“某一侧能看、另一侧不能看”的割裂。

## P2：收紧 fixture 回退策略

- [ ] 重新评估市场页的 fixture 自动回退策略，避免把“真实接口无数据”和“本地示例兜底”混在同一层展示里。
- [ ] 为市场页增加更显式的 degraded mode 提示，让用户能明确判断当前看到的是示例数据还是真实数据。
- [ ] 评估是否为生产/联调环境增加关闭 fixture 回退的开关。

## 暂不纳入当前待办

- 真实券商接入
- 多账户体系
- 多市场数据源
- 多策略组合
- AI / ML 训练与实验平台
